## What Is Chunking?

Chunking is the process of splitting a long document into small, overlapping
text segments before embedding and indexing them.

A long document cannot be indexed as one unit. If we embedded the entire
medical_knowledge.md as a single vector, that vector would need to represent
fever, headache, cough, cold, dehydration, prevention, and warning signs all
at once. The resulting embedding would be an average over all topics.

When a user asks about headaches, the whole-document vector would score only
moderately -- because most of the document is not about headaches. The
specific paragraph about headache causes would score much higher if it were
indexed as a separate vector.

Chunking solves this by giving each passage its own focused vector.

---

## Why Overlap Matters

Overlap means repeating some text at the start of each new chunk.

Without overlap, the split might look like:

    Chunk 1: "...dehydration is a well-known trigger for headaches."
    Chunk 2: "When the body does not have enough fluid, brain tissue..."

Chunk 2 starts mid-explanation. A user asking about headaches and dehydration
might retrieve chunk 2 and miss the context from chunk 1.

With 80 characters of overlap:

    Chunk 1: "...dehydration is a well-known trigger for headaches."
    Chunk 2: "trigger for headaches. When the body does not have enough fluid..."

Now chunk 2 carries enough context to be understood on its own.

---

## How Chunk Size Affects Retrieval Quality

| Chunk Size | Effect |
|---|---|
| Too small (< 150 chars) | Chunks lack context. Sentences are meaningless without surrounding text. |
| Good range (300-600 chars) | Focused on 2-4 sentences. Captures one idea clearly. |
| Too large (> 800 chars) | Covers multiple ideas. Embedding is diluted. Retrieval precision drops. |

This project uses 400 characters with 80 characters of overlap.

## How Metadata Is Stored

Each chunk carries:

    chunk_id:    "fever-chunk-002"         -- unique ID for tracking
    text:        "Fever is a temporary..."  -- the actual passage text
    topic:       "fever"                   -- section it came from
    source_file: "medical_knowledge.md"    -- origin document
    char_start:  0                         -- position in source document
    char_end:    395                       -- end position in source document
    chunk_index: 2                         -- position within its topic

FAISS stores only the vectors. Metadata is saved separately in
vector_store/metadata.json, aligned by integer position.

## Why This Is a Major Step Toward Real RAG

A complete RAG pipeline:

    Document -> Chunker -> Embedder -> Vector Index
                                            |
    User query -> Embedder -> Search Index -> Top-K Chunks
                                                     |
                              Prompt Builder -> [Chunk 1, Chunk 2, Chunk 3]
                                                     |
                                              LLM -> Generated Answer

Day 3 completes every step except the last two (Prompt Builder and LLM).
The all_chunks list returned by the retriever is exactly what will be
passed to the LLM in Day 4.

## Remaining Limitations

- The system still returns raw chunk text as the answer. There is no LLM to
  synthesise multiple chunks into a fluent, complete response.
- A query like "why do I feel hot and dizzy?" may retrieve relevant chunks from
  both fever and dehydration, but the user only sees the single best chunk.
  Day 4 will use all retrieved chunks as context for LLM generation.
- Chunk size is fixed. Real production systems adapt chunk size to the content
  type -- smaller chunks for dense technical text, larger for narrative text.

---

## Why Day 4 Adds Prompt Building and LLM Generation

After retrieval, the system has a list of relevant text passages. Without an
LLM, the best it can do is return the top passage verbatim. This is retrieval,
not generation.

An LLM receiving all top-k chunks as context can:
- Synthesise information from multiple chunks into one coherent answer.
- Rephrase the retrieved text to directly address the user's specific question.
- Handle queries that require combining two or more topics.
- Generate a response in natural, conversational language.

Day 4 will add a prompt builder that assembles the retrieved chunks into a
structured prompt, and connect it to a local or API-based LLM for answer generation.
